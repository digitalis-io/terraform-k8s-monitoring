## [Unreleased]

### Added

- `modules/mimir`: `create_namespace` variable (default `true`) â€” set to `false` to deploy Mimir into a namespace managed elsewhere (bring-your-own-namespace), matching the existing behaviour of `modules/loki`. Previously the module always created its namespace, so `apply` failed when the namespace already existed. Fixes [#24](https://github.com/digitalis-io/terraform-k8s-monitoring/issues/24).
- `modules/mimir`: `storage.gcs_blocks_prefix`, `storage.gcs_ruler_prefix`, and `storage.gcs_alertmanager_prefix` variables â€” object key prefixes (rendered as `storage_prefix`) for the blocks/ruler/alertmanager GCS backends, letting Mimir share one GCS bucket across components (each storage type needs a distinct bucket+prefix combo). Empty (default) omits the prefix.
- `modules/otel-collector`: `mimir2_endpoint` and `mimir2_tenant_id` variables â€” a second, independent Prometheus remote-write target (`prometheusremotewrite/mimir2`) so metrics can dual-write to two backends at once without one exporter replacing the other. Empty (default) disables the second target.
- `modules/otel-collector`: `logs_gigapipe_label_allowlist` variable (list of resource-attribute keys, default `[]`) â€” when set alongside `otlphttp_logs_endpoint`, logs to Gigapipe run a dedicated pipeline that `keep_keys()`-trims resource attributes to this allowlist (Gigapipe indexes every attribute as a label, unlike Loki's metadata tier), so Gigapipe and Loki index the same label set. `[]` (default) sends all attributes.
- `modules/otel-collector`: `otlp_grpc_max_recv_msg_size_mib` variable (default `4`) â€” raises the OTLP gRPC receiver's max receive message size beyond gRPC's 4 MiB default for producers sending large post-decompression batches.
- `modules/tempo`: `extra_values` variable â€” raw Helm values YAML merged on top of the module's generated values (e.g. per-component `nodeSelector`/`tolerations`, for which the tempo-distributed chart has no single global knob).
- `modules/alloy`: `ebpf_profiling` block (`enabled`, `collect_interval`, `demangle`) â€” discovers pods annotated `profiles.grafana.com/cpu.scrape: "true"` and profiles them via the `pyroscope.ebpf` component, pushing to `pyroscope_endpoint`. Requires `pyroscope_endpoint` set and `controller_type = "daemonset"` (validated); sets `hostPID: true` and `securityContext.privileged: true` when enabled.
- `modules/alloy`: `node_selector` and `tolerations` variables for pod scheduling (the module previously had no way to schedule onto tainted node pools, e.g. GKE's `kubernetes.io/arch=arm64:NoSchedule`).
- `modules/otel-collector`: `otlphttp_logs_endpoint` and `otlphttp_traces_endpoint` variables â€” generic OTLP/HTTP exporters (endpoint appends `/v1/logs` or `/v1/traces` itself) for backends that don't match the Loki (`<endpoint>/otlp`) or Tempo (gRPC `:4317`) conventions already supported, e.g. Gigapipe/qryn's native `/v1/logs`, `/v1/traces` routes.
- `modules/otel-collector`: `metrics_collection_interval` variable â€” uniform override for the hostmetrics/kubeletstats/k8s_cluster receivers, which otherwise default to different intervals (10s/20s/30s).
- `modules/otel-collector`: `prometheus_scrape_targets` variable (list of `host:port`, default `[]`) â€” adds an opt-in `prometheus` receiver (mode `deployment` only) with a static `scrape_configs` job (`kube-state-metrics`) targeting those hosts, wired into the metrics pipeline alongside `k8s_cluster`. Lets consumers scrape a Prometheus-exposition-format source (e.g. kube-state-metrics) so downstream Prometheus-remote-write/kube-prometheus-stack-style dashboards get real `kube_*`-named metrics instead of relying solely on the OTel-semantic-convention names from `k8s_cluster`/`kubeletstats`.
- `modules/cert-manager` and `modules/otel-collector`: `node_selector` and `tolerations` variables for pod scheduling. `tolerations` lets the pods schedule onto tainted node pools (e.g. GKE's automatic `kubernetes.io/arch=arm64:NoSchedule` taint on Arm pools). cert-manager applies both to all components (controller, webhook, cainjector, startupapicheck); otel-collector applies them to the collector and, when `operator.enabled`, to the operator manager pod. For `Exists`-style tolerations set `operator = "Exists"` and leave `value` empty.
- `modules/otel-collector`: `clickhouse_cluster` and `clickhouse_table_engine` variables. Set `clickhouse_cluster` to emit `ON CLUSTER` DDL (with `clickhouse_table_engine` e.g. `ReplicatedMergeTree`) so `otel_logs`/`otel_traces` are created on every node of a load-balanced ClickHouse cluster, not just the connected one.
- `modules/otel-collector`: `release_name` variable (default `"otel"`) to override the Helm release name â€” lets you run two collectors in one namespace (e.g. a `deployment` gateway plus a `daemonset` agent).
- `modules/cert-manager`: configurable ClusterIssuer via a new `issuer` object variable. `issuer.type` selects `self_signed` (default; unchanged behaviour), `acme` (Let's Encrypt / any ACME CA with an HTTP-01 ingress solver â€” set `issuer.acme.email`, optionally `server`/`private_key_secret`/`solver_ingress_class`), or `ca` (sign off an existing CA keypair in `issuer.ca.secret_name`). Previously the module could only ever create a self-signed issuer, which browsers reject for public ingress TLS. New `cluster_issuer_manifest` output exposes the rendered manifest. Fixes [#22](https://github.com/digitalis-io/terraform-k8s-monitoring/issues/22).
- `modules/prometheus`: `grafana_replicas` variable (default `1`) to scale Grafana horizontally. Values > 1 require `grafana_database` (validated) â€” SQLite cannot be shared across pods.
- `modules/prometheus`: external database backend for Grafana via new `grafana_database` object variable (`type` `postgres`/`mysql`, `host`, `name`, `user`, plus `ssl_mode` for postgres). Moves Grafana state off the default ephemeral SQLite â€” required to run more than one Grafana replica. The password is supplied either as plaintext `password` (the module creates a `prometheus-grafana-db` Secret) or via `password_secret` referencing an existing Secret; it is injected as `GF_DATABASE_PASSWORD` from a `secretKeyRef`, never rendered into the Helm values. Leave `grafana_database` null (default) to keep the chart's built-in SQLite.
- `modules/prometheus`: per-component enable/disable toggles â€” `prometheus_enabled`, `prometheus_operator_enabled`, `kube_state_metrics_enabled`, `node_exporter_enabled`, and `default_rules_enabled` (all default `true`). Lets you slim the kube-prometheus-stack deployment to any subset of components â€” e.g. set all five to `false` with `grafana_enabled = true` for a Grafana-only install. CRDs are always installed by the chart and are unaffected by these toggles.
- `modules/tempo`: `storage.s3_key_prefix` variable â€” object key prefix for all Tempo S3 objects, letting Tempo share one S3 bucket with Mimir/Loki without key collisions (fixes #18).
- `modules/pyroscope`: `storage.s3_key_prefix` variable â€” object key prefix (`structuredConfig.storage.prefix`) for all Pyroscope S3 objects, letting Pyroscope share one S3 bucket with other components.
- `modules/prometheus`: Tempoâ†’Pyroscope **Trace to profiles** correlation. The Tempo Grafana datasource now gets a `tracesToProfiles` block linking a span to its CPU profile in Pyroscope (matched by `service.name`). New `tempo_profile_type_id` variable (default `"process_cpu:cpu:nanoseconds:cpu:nanoseconds"`) selects the default profile type; set `""` to disable. Active only when both `tempo_datasource_url` and `pyroscope_datasource_url` are set.
- `modules/prometheus`: Lokiâ†’Tempo logâ†”trace correlation. The Loki Grafana datasource now gets a `derivedFields` entry that links each log line's trace id to the trace in Tempo (a **View trace** link), complementing the existing Tempoâ†’Loki `tracesToLogsV2` wiring. The Tempo datasource is given a stable `uid: tempo`. New `loki_trace_id_field` variable (default `"trace_id"`) names the trace-id field in the JSON log body; set `""` to disable. Active only when both `loki_datasource_url` and `tempo_datasource_url` are set.
- `modules/otel-collector`: `log_parsing` object variable to configure structured-log parsing on the daemonset `filelog` receiver. New fields `json_enabled`, `json_match_expr`, `severity_field`, `trace_enabled`, `trace_id_field`, `span_id_field` allow you to match your application's log field names and disable parsing entirely if needed. Defaults preserve the built-in JSON detection (`^\\s*[{]`) and field names (`trace_id`, `span_id`, `level`).
- `modules/alloy`: new module â€” Grafana Alloy collector (daemonset/deployment/statefulset) with River/Alloy pipeline config, sibling-module integration hooks for Loki, Tempo, Mimir, Pyroscope, and OTel Collector, and WAL persistence support for statefulset mode.
- `modules/alloy`: opt-in `faro_receiver` block for ingesting [Grafana Faro](https://grafana.com/oss/faro/) real-user-monitoring (RUM) telemetry (JS errors, traces, logs, web-vitals). Grafana does not publish a standalone "faro" Helm chart â€” enabling `faro_receiver.enabled` swaps the built-in default config to Alloy's `faro.receiver` component (logs â†’ Loki, traces â†’ Tempo; no Mimir wiring, faro.receiver has no metrics output). Enabling `ingress` now requires `faro_receiver.enabled = true` and sets the chart's `ingress.faroPort` explicitly â€” the chart's ingress feature always targets a fixed Faro port regardless of any other component, so ingress on a non-Faro release was previously dead configuration. New `release_name` variable (default `"alloy"`) lets a second Alloy-based release â€” e.g. this Faro receiver â€” be deployed into the same namespace as a daemonset collector without colliding; new `extra_ports` variable overrides the default OTLP ports (auto-defaults to the Faro port when `faro_receiver.enabled = true`); new `helm_release_id`, `faro_receiver_http_endpoint`, and `faro_receiver_public_url` outputs.
- `modules/alloy`: sensitive-data (PII) processor, **enabled by default**, hashing or deleting known-sensitive attribute names on logs, traces, and metrics before export (`otelcol.processor.transform`, per [OTel's sensitive-data guidance](https://opentelemetry.io/docs/security/handling-sensitive-data/)). Default financial-institution field list covers credit card numbers, CVV, passwords/secrets/tokens, SSNs, IBANs/bank accounts, and email addresses. Configure via the new `sensitive_data` variable (`enabled`, `action`, `default_rules_enabled`, `custom_rules`, `salt`). Only wired into the built-in config; a non-empty `alloy_config` must handle redaction itself.
- `examples/alloy-basic`: minimal example wiring Alloy as a daemonset collector forwarding OTLP signals to Loki, Tempo, and Mimir.
- `examples/alloy-faro-receiver`: example deploying a daemonset OTLP collector alongside a second Alloy release (`release_name = "faro-receiver"`, `controller_type = "deployment"`, `faro_receiver.enabled = true`) accepting RUM telemetry from the Faro Web SDK, both wired to Loki and Tempo.
- `modules/otel-collector`: ClickHouse exporter support via new `clickhouse_endpoint`, `clickhouse_username`, `clickhouse_password`, `clickhouse_database`, and `clickhouse_create_schema` variables. Logs and traces can now be forwarded to ClickHouse alongside (or instead of) Loki/Tempo.
- `modules/otel-collector`: OpenTelemetry Operator support via new `operator` object variable (opt-in; `operator.enabled` defaults to `false`). When enabled, deploys the OTel Operator Helm chart which provides `OpenTelemetryCollector` and `Instrumentation` CRDs. Includes optional Go eBPF auto-instrumentation support (`operator.go_instrumentation_enabled`; requires Linux kernel â‰Ą 4.19).
- `modules/otel-collector`: Host metrics collection for daemonset mode â€” `hostmetrics` receiver added to the metrics pipeline when `mode = "daemonset"`.
- `modules/otel-collector`: Kubernetes metrics collection. `mode = "daemonset"` now also runs the `kubeletstats` receiver (per-node pod/container/node/volume CPU, memory, network, and filesystem from the local kubelet on `:10250`), and `mode = "deployment"` runs the cluster-singleton `k8s_cluster` receiver (object-state metrics: pod phase, node conditions, deployment/replica/quota counts). The `clusterRole` gains the matching RBAC (`nodes/stats`, `nodes/proxy` for daemonset; broad object read for deployment). Closes the gap where only node-OS metrics â€” not Kubernetes workload/cluster metrics â€” were collected.
- `modules/otel-collector`: `mimir_tenant_id` variable to set the `X-Scope-OrgID` header on remote-write requests, matching the Mimir multi-tenancy configuration.
- `modules/prometheus`: ClickHouse datasource support in Grafana via new `clickhouse_datasource` object variable. Configures the `grafana-clickhouse-datasource` plugin with OTel schema table references for logs and traces.
- `modules/prometheus`: `grafana_plugins` variable to control which Grafana plugins are installed. Defaults include common community plugins; ClickHouse datasource plugin included.
- `modules/tempo`: `metrics_generator_remote_write_url` variable to enable the Tempo metrics generator for `rate()` and span metrics. Set to the Mimir remote-write URL to activate; empty string disables the generator (default).

### Fixed

- `modules/prometheus`: `grafana_database` null-safety on OpenTofu < 1.11 / Terraform. Older engines eagerly evaluate both sides of `&&`/`||`, so the `grafana_database == null || â€¦` and `grafana_database != null && â€¦` guards still errored with "Attempt to get attribute from null value" whenever `grafana_database` was unset (the default). The five variable validations and the `grafana_db_create_secret` local are now wrapped in `try(â€¦)`, so a plan with the default (no external DB) succeeds on OpenTofu 1.9.x.

### Changed

- `modules/cert-manager`: the ClusterIssuer `terraform_data` now re-applies when the issuer type/config **or** name changes (previously only a chart-id change re-triggered it), and `cluster_issuer_name` is validated as an RFC 1123 DNS subdomain. The manifest is built with `yamlencode` rather than string interpolation, removing a YAML/shell-injection surface via the issuer name.
- `modules/otel-collector`: Chart version updated from `0.150.0` to `0.158.2`.
- `modules/otel-collector`: Default resource requests increased from `100m CPU / 128Mi` to `300m CPU / 256Mi` to support hostmetrics collection alongside traces and logs.
- `modules/prometheus`: Chart version (`kube-prometheus-stack`) updated from `75.2.0` to `86.3.2`.
- `modules/prometheus`: Grafana `tracesToMetrics.datasourceUid` corrected from `"Mimir"` (display name) to `"mimir"` (UID) â€” fixes broken request-rate and latency links in the Tempo trace view.
- `modules/otel-collector`: `helm_release` values wrapped in `sensitive()` to suppress ClickHouse credentials from plan output. Credentials remain in Terraform state; ensure your state backend encrypts at rest.
- `modules/prometheus`: `helm_release` values wrapped in `sensitive()` to suppress ClickHouse datasource password from plan output.
- `modules/mimir`: `ingress_host` validation now enforces RFC 1123 hostname format in addition to requiring a non-empty value when `ingress_enabled = true`.
- `modules/prometheus`: `grafana_ingress.host`, `prometheus_ingress.host`, and `alertmanager_ingress.host` validation now enforces RFC 1123 hostname format when the respective ingress is enabled.

### Security

- `modules/grafana-rules`: contact points (which embed the Slack webhook URL and PagerDuty integration key) are now provisioned as a Kubernetes **Secret** instead of a ConfigMap. ConfigMaps are stored in plaintext and have weaker RBAC than Secrets; the Grafana alerting sidecar watches both resource types (chart default `sidecar.alerts.resource = "both"`), so provisioning is unaffected.
- `modules/prometheus-rules`: the Slack webhook URL and PagerDuty routing key are no longer stored raw in the AlertmanagerConfig `terraform_data.triggers_replace` (a non-sensitive attribute that echoed them in plan output). They are hashed with `sha256` â€” the config still re-applies when a credential changes. The credentials themselves continue to live only in Kubernetes Secrets.

### Fixed

- `modules/loki`: S3 object storage via an existing credentials Secret (`storage.s3_credentials_secret`) never authenticated. The Helm values render `accessKeyId: $${AWS_ACCESS_KEY_ID}` (and secret key) as env-var references, but the grafana/loki chart does not start Loki with `-config.expand-env=true`, so Loki read those as literal strings and S3 auth failed silently. The module now adds `-config.expand-env=true` to the `extraArgs` of every enabled component (`singleBinary`/`read`/`write`/`backend`) whenever an S3 credentials Secret is used. (mimir-distributed already sets this flag on every component, so Mimir was unaffected.) Fixes [#23](https://github.com/digitalis-io/terraform-k8s-monitoring/issues/23).
- `modules/prometheus-rules`: the AlertmanagerConfig now `depends_on` the Slack/PagerDuty Secrets it references by name, so they are guaranteed to exist before it is applied.
- `modules/loki`: the `storage` block now explicitly nulls out every backend sub-block (`s3`/`gcs`/`azure`/`filesystem`) except the selected `storage_backend`. The chart's own defaults populate all backend blocks and Helm's value merge does not drop the unset ones, so `loki.commonStorageConfig` saw multiple backends "configured" at once and refused to start ("too many storage configs provided in the common config").
- `modules/alloy`: the `controller:` Helm values block (`type`/`hostPID`/`nodeSelector`/`tolerations`) was emitted nested inside `alloy:` instead of as a top-level sibling key. The chart reads `.Values.controller`, not `.Values.alloy.controller`, so Helm silently dropped all of it regardless of what was set â€” `controller_type`, `replicas`, and (once added) the new `node_selector`/`tolerations`/`ebpf_profiling` hostPID setting never actually applied. Moved the block to the end of the template, after `ingress:` (which was already correctly top-level), matching the chart's real values schema.
- `modules/alloy`: the `sensitive_data` OTTL statements already contain real `"` characters (e.g. `delete_key(attributes, "token")`), but the template re-wrapped each one in another unescaped `"${statement}"`, producing invalid River syntax (`missing ',' in expression list`) whenever `sensitive_data.enabled` was left at its default `true`. Now escaped via `replace(statement, "\"", "\\\"")` at each of the three (log/trace/metric) statement lists.
- `modules/prometheus`: fixed the Lokiâ†’Tempo **View trace** derived field opening Tempo with an empty query. Two causes: (1) `matcherType: label` does not resolve a value for OTLP structured-metadata fields in the Grafana Logs Drilldown app â€” switched to a regex matcher against the JSON log body (`"<field>":"(\w+)"`); and (2) the derived-field `url` was emitted as `${__value.raw}`, which Grafana's provisioning-file `$`-interpolation consumes at load time (storing an empty `url`) â€” it is now emitted as `$${__value.raw}` (a triple-`$` in the template, which `templatefile()` collapses to the required double-`$`) so Grafana stores the literal `${__value.raw}` and substitutes the captured trace id at click time. Fixes [#12](https://github.com/digitalis-io/terraform-k8s-monitoring/issues/12).
- `modules/otel-collector`: the `filelog` receiver (daemonset mode) now parses structured JSON pod logs and promotes `trace_id`/`span_id` into native OTel fields. Guarded JSON and trace parsers populate ClickHouse `otel_logs.TraceId`/`SpanId`/`SeverityText`, enabling native logâ†”trace correlation without `JSONExtractString(Body, â€¦)` and supporting Grafana logsâ†”traces linking. Fixes [#10](https://github.com/digitalis-io/terraform-k8s-monitoring/issues/10).
- `modules/alloy`: a custom `alloy_config` override rendered with its first line un-indented, producing an invalid YAML block scalar (`content: |`). `indent()` skips the first line by design; the template now indents it explicitly. Only affects users who set `alloy_config` directly â€” the built-in default config path was unaffected.
- `modules/alloy`: `otlp_grpc_endpoint` and `otlp_http_endpoint` outputs hardcoded the `alloy.` release name instead of reading the actual `release_name`, producing wrong endpoints for any release deployed under a custom name.
- `modules/alloy`: `otel_grpc_endpoint` was defined and rendered an `otelcol.exporter.otlp "upstream"` block, but the block was never wired into the batch processor's `output {}` â€” setting it had no effect. Metrics, logs, and traces in the built-in OTLP-receiver config now fan out to any combination of `mimir_endpoint`/`loki_endpoint`/`tempo_endpoint` (direct) and `otel_grpc_endpoint` (upstream OTel Collector), so a caller can pick Mimir/Loki/Tempo-only, OTel Collector-only, or dual-write to both.
- `local-exec` kubectl provisioners in `modules/cert-manager` and `modules/prometheus-rules` no longer hardcode `$HOME/.kube/config` as the kubeconfig fallback. When `kubeconfig_path` is empty, `--kubeconfig` is omitted entirely so kubectl uses its standard resolution order (`KUBECONFIG` env var â†’ `~/.kube/config`). Fixes terraform apply failures when `$HOME/.kube/config` does not exist.
